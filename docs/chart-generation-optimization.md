# チャート画像生成バッチの高速化計画書

本ドキュメントは、Yahoo Finance からの株価データ取得に伴う「全銘柄のチャート画像生成バッチ（`generate-batch`）」について、サーバースペックおよびソースコードの調査結果からボトルネックを特定し、高速化するためのアプローチをまとめたものです。

---

## 1. サーバー環境スペック (System Hardware Spec)

調査により判明したホストサーバーのハードウェア仕様は以下の通りです。

| 項目 | スペック | 備考 |
| :--- | :--- | :--- |
| **CPU** | Intel(R) Core(TM) i5-7260U CPU @ 2.20GHz | 2コア / 4スレッド |
| **メモリ (RAM)** | 15GiB (約 16GB) | 空き容量 約9GiB（十分に余裕あり） |
| **ストレージ I/O** | SSD (NVMe / SATA) | ディスク性能は十分だが、同期書き込みはボトルネック要因 |

## 2. 現状のパフォーマンスとボトルネック分析

### 現状の測定値 (Before)
* **総処理数**: 3,844 銘柄（「日足」「週足」「月足」の合計 11,532 枚の WebP 画像）
* **処理速度**: 平均 **2.2 銘柄/秒**（画像ベースで約 6.6 枚/秒）
* **合計所要時間**: **1,724.1 秒** (約 **28.7 分**)

### 特定された主なボトルネック
1. **価格データアクセスのシリアル実行 (I/Oブロック)**:
   [src/chart-generator/batch.ts](file:///home/oharato/workspace/stock-data/src/chart-generator/batch.ts#L86-L117) において、各銘柄の中で日足・週足・月足のデータを DuckDB から取得する際、`await` によって直列（シーケンシャル）に実行されています。
   これにより、DuckDB へのクエリ発行の待ち時間が3回分累積しています。
2. **同期ディスク書き込み (イベントループのブロック)**:
   生成した WebP 画像バイナリを `writeFileSync` を用いて同期的に書き込んでいます。I/O 待ちの間、Node.js のシングルスレッドが完全にブロックされ、並行処理の効率が低下します。
3. **インクリメンタル生成（差分更新）の欠如**:
   前回のバッチ実行で生成済みの画像であっても、バッチ実行のたびに全 3,844 銘柄の全画像を再生成しています。実際には、株価が更新された銘柄（または直近で動いた銘柄）のみを生成すれば十分です。

## 3. 高速化に向けた 5 つの最適化アプローチ

### 🚀 アプローチ 1: DuckDB データアクセスの並行化 (`Promise.all`)
**対策**:
日足・週足・月足のデータ取得はそれぞれ独立しているため、`Promise.all` を使用して並行実行します。
これにより、DBアクセスの往復遅延を **約3分の1** に短縮します。

```typescript
// 改善前 (直列実行)
const dailyData = await fetchDailyPrices(code, 120, tempDb);
const weeklyData = await fetchWeeklyPrices(code, 100, tempDb);
const monthlyData = await fetchMonthlyPrices(code, 120, tempDb);

// 改善後 (並行実行)
const [dailyData, weeklyData, monthlyData] = await Promise.all([
  fetchDailyPrices(code, 120, tempDb),
  fetchWeeklyPrices(code, 100, tempDb),
  fetchMonthlyPrices(code, 120, tempDb),
]);
```

### 🚀 アプローチ 2: 非同期書き込み（`fs/promises`）の採用
**対策**:
`writeFileSync` を `fs.promises.writeFile` に置き換えます。ファイル書き込みをスレッドプールに逃がすことで、Node.js のメインスレッドをブロックせず、次の銘柄の描画ロジックを即座に開始します。

### 🚀 アプローチ 3: 差分更新（インクリメンタルビルド）の導入
**対策**:
以下の判定ロジックをバッチ処理の最初に追加します。
1. 各銘柄の 3 つのチャート画像が存在するか確認。
2. 存在する場合、画像の最終更新日時とデータベースの最新株価日付（`prices` テーブルの当該 ticker の最大日付）を比較。
3. 画像が最新日付以降に生成されたものであれば、画像生成を **スキップ** する。

これを行うことで、日々の定期更新バッチにかかる時間は **数分から数秒** に激減します。

### 🚀 アプローチ 4: 並列度（Concurrency）の再調整
**対策**:
現在の並列度制限 `CONCURRENCY_LIMIT = 5` を **`10` 前後** に引き上げます。
I/O 待ちが非同期化されることでスレッドの遊び時間が減るため、CPU スレッド数（4スレッド）に対して高めの並列度を設定した方がスループットが向上します。

### 🚀 アプローチ 5: `sharp` のスレッドプール調整
**対策**:
`sharp.concurrency(1)` を設定して、画像処理エンジン（libvips）の内部マルチスレッドを 1 に制限します。
JS 側が並列（`p-limit`）で動いているため、各 `sharp` インスタンスがスレッドを奪い合うのを防ぎ、コンテキストスイッチのオーバーヘッドを削減します。

## 6. 結論と考察

1. **実運用（デイリー更新）における圧倒的な効率化**:
   差分更新ロジックの導入により、バッチはDBの更新状況を検知して不要な描画を自動的にスキップします。これにより、2回目以降のバッチ実行時間は **0.6秒** となり、リソース消費をほぼゼロに抑えることに成功しました。
2. **フル生成時のCPU限界（ボトルネックの正体）**:
   データアクセスの並行化やファイル書き込みの非同期化を行っても、フル生成時の処理速度は `2.20 t/s` ➔ `2.27 t/s` と約3%の向上に留まりました。
   これは、SVG文字列の生成と `sharp` (libvips) による WebP 画像圧縮エンコードという **「CPUバウンドの計算処理」** 自体が最大のボトルネックであり、2コア/4スレッドの CPU リソースをすでに限界（使用率 100%）まで使い切っているためです。
   I/O ボトルネックは元々小さく、計算資源が処理上限を決定づけていることが今回の計測で実証されました。
